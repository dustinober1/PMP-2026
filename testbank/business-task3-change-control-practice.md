## Key Takeaways: Change Control Pro-Tips

1. **Analyze Before Acting**: Most Monitoring & Controlling questions follow the pattern: confirm variance → analyze impact/root cause → choose action → submit change request if baseline affected → implement and communicate. "Doing work" without the decision step is often the wrong answer.

2. **Baseline Integrity First**: Approved changes must update ALL affected baselines (scope, schedule, cost) before implementation proceeds. Partial updates create false project reality. In predictive projects, even approved changes await formal Work Authorization before implementation begins.

3. **Closed-Loop Communication**: Change requesters deserve notification of decisions. Silence is not an acceptable outcome in change control. The change control process must include formal communication mechanisms to ensure requesters receive timely status notifications.

4. **Verbal ≠ Approved**: Even well-intentioned verbal approvals require formal documentation. In regulated environments, written authorization is the only reliable evidence of change control decisions. Verbal directions become disputed memories; documentation protects the project.

5. **Hybrid Governance Boundaries**: In hybrid projects, distinguish between changes within team discretion (backlog prioritization within fixed scope/schedule/cost) vs. baseline-impacting changes requiring CCB approval. The governance layer approves what gets done; the delivery layer decides when and how.

6. **Configuration Alignment**: When baselines change, ALL project documents must be synchronized. Scope statement, WBS, risk register, and quality plans must reflect the same reality. Operating with misaligned documentation creates team confusion and corrupts performance measurement.

| **PMI Mindset** | The PM enforces change control processes while being pragmatic about implementation. The goal is to stop the bleeding and address the damage, not to create more disruption through punitive rollback. |

### Question 49

<strong>Enabler:</strong> 3 | <strong>Methodology:</strong> Hybrid

<strong>Scenario:</strong>
CR-128 was approved with a condition: the change must be implemented within the current fiscal quarter to align with budget cycles. During sprint execution, the team realizes they cannot complete the work by quarter-end due to a critical dependency on another project that is delayed. The fiscal year constraint is firm, and delaying implementation would require waiting until next year.

<strong>Question:</strong>
What is the MOST APPROPRIATE action for the project manager to take?

- <strong>Correct:</strong> Immediately notify the CCB that the conditional approval cannot be met and request direction on whether to defer or cancel the change
- <strong>Distractor 1:</strong> Ask the team to work overtime to complete the implementation by the quarter deadline
- <strong>Distractor 2:</strong> Proceed with implementation and accept that the budget impact will spill into the next fiscal year
- <strong>Distractor 3:</strong> Pause implementation and resume at the start of the next fiscal year without further CCB involvement

<details>
<summary>Detailed Remediation</summary>

| Aspect | Details |
| :--- | :--- |
| **Core Logic** | Conditional approvals are based on specific assumptions. When those assumptions prove false, the approval itself is invalidated. The CCB must reconsider the change given the new reality, as timing constraints were material to their decision. |
| **PMI Mindset** | The PM provides transparent communication when approval conditions cannot be met. The governance body retains decision authority; the PM does not work around constraints through overtime or unilateral timeline extensions. |
| **The Trap** | Overtime or spilling budget seems like implementation commitment, but violates the approval conditions. Automatic deferral assumes the CCB's intent, which may not align with waiting a full fiscal year. |

### Question 50

<strong>Enabler:</strong> 4 | <strong>Methodology:</strong> Hybrid

<strong>Scenario:</strong>
A project is closing, and the project manager is preparing the final report. The report must account for all approved change requests and their impact on the original baselines. The project manager discovers that CR-095, approved six months ago, was never implemented. The budget for CR-095 was spent, but no value was delivered.

<strong>Question:</strong>
What is the BEST action for the project manager to take?

- <strong>Correct:</strong> Document CR-095 as an unused approved change in the final report, return any unspent funds, and include lessons learned about change tracking
- <strong>Distractor 1:</strong> Implement CR-095 immediately before project closure to deliver the approved value
- <strong>Distractor 2:</strong> Exclude CR-095 from the final report since it was never implemented and including it would reflect poorly on project management
- <strong>Distractor 3:</strong> Transfer the CR-095 budget to other approved changes that ran over cost to balance the final budget

<details>
<summary>Detailed Remediation</summary>

| Aspect | Details |
| :--- | :--- |
| **Core Logic** | Project closure requires transparency about all approved changes, whether implemented or not. Hiding the failure or rushing implementation at closure undermines the closure process and accurate historical records. The closure report should capture reality, including process failures. |
| **PMI Mindset** | The PM provides complete and accurate information in closure reports, even when it reflects poorly on project management. Lessons learned from failures are valuable organizational assets. |
| **The Trap** | Implementing at closure seems to "fix" the problem but rushes work without proper execution discipline. Excluding the failure or hiding budget transfers compromises the integrity of closure documentation. |
