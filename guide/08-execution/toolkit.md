## üß± Impediment / Blocker Triage (60-Second Checklist)
Use this when a blocker surfaces during a standup or checkpoint:

1. **Clarify**: What is blocked and what is the smallest next step?
2. **Impact**: Who is blocked and for how long? What deliverable is at risk?
3. **Owner**: Who can fix it (team, vendor, functional manager, IT)?
4. **Escalation path**: Who has authority if the owner can‚Äôt resolve quickly?
5. **Decision**: What do you need (access, approval, funding, priority change)?
6. **Follow-up**: When will you check back (today, not next week)?

## üîÅ Change Request ‚ÄúMini-Analysis‚Äù (Fast Template)
Use for predictive change control, or for hybrid governance decisions.

| Field | Example |
|---|---|
| **Request** | Add SSO login option |

::: info üîç PMP nuance
In agile delivery, ‚Äúchange‚Äù often means **backlog re-prioritization**. In predictive delivery, it often means **formal integrated change control**. In hybrid, you may do both.

## üìå RAID Log (Risks, Actions, Issues, Decisions)
RAID is a simple ‚Äúsingle source of truth‚Äù for execution.

| Type | What it captures | Example |
|---|---|---|
| **Risk** | Uncertain event | Vendor may miss delivery date |
| **Action** | Planned work to reduce risk / improve outcome | Run pilot with 10 users |
| **Issue** | Current problem | Test environment unstable |
| **Decision** | A recorded choice and rationale | Choose AWS over Azure due to compliance |

::: tip üí° Exam shortcut
When a scenario has confusion and rework, it‚Äôs usually missing a **decision log** (people keep re-litigating old decisions).
:::

## üì£ One-Page Status Report (Template)
Use this structure for executives and busy stakeholders:

- **Overall status**: Green / Yellow / Red (with 1-sentence reason)
- **This period**: Key accomplishments
- **Next period**: Top planned work
- **Risks**: Top 3 (owner + response)
- **Issues**: Top 3 (owner + next action)
- **Decisions needed**: 1‚Äì3 items (deadline + options)
- **Value signal**: Adoption/usage, demo feedback, defect trend (pick what matters)

## üèÅ Transition to Operations (Operational Readiness Checklist)
Use before go-live/handoff to keep value from collapsing after delivery.

- **People**: Training complete, champions/support staffed, roles/RACI confirmed
- **Process**: SOPs updated, escalation path defined, SLAs agreed, incident process ready
- **Technology**: Monitoring/alerts set, backups/rollback plan, access/provisioning ready
- **Knowledge**: Runbooks + troubleshooting docs available, known issues documented
- **Benefits**: Benefits owner confirmed, measurement cadence scheduled
